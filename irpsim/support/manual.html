1. Object Definition Language Overview

IRPSIM’s Object Definition Language (ODL) is used to define all variables, scripts and other objects that IRPSIM uses to run a simulation. With the exception of Definitions and Options and Regions, an ODL object is defined in a block of text that begins with #ObjectType name and ends with #END. ObjectType may be one of the following:

  • VARDEF

  • SCRIPT

  • SCENARIO

  • CATEGORY

Examples
#VARDEF basic_apportionment (1)
#type          supply	sum
#region	cra
(basic_apportionment_array[_cyear-(year0)][_ctyear-tyear0])
#ENDVAR (2)

#SCRIPT	basemix-cra-ondemand (1)
...
Body of script
...
#END (2)

#CATEGORY groundwater_blended (1)
#allocationmethod BalanceTake
chino compton_cup elsinore foothill_cup lakewood live_oak long_beach orange_county upper_claremont
#END (2)
1 Object name
2 Closing #END tag
The ODL is case-insensitive so, for example, #VARDEF may also be written as #vardef
Previous versions of IRPSIM required #VARDEF blocks to be terminated by an #ENDVAR tag. IRPSIM Version 9 continues to accept #ENDVAR tags, but you may also use the shorter #END tag.

1.1. Comments

Any line in an ODL file that begins with an asterisk is considered a comment and will be ignored by IRPSIM when it loads the file.

* This line is a comment

*****------------------ So is this line ---------------****

# Not a comment (in fact, this line would generate an error)

1.2. Including Files

Normally, IRPSIM parses an ODL file by reading one line at a time until it reaches the end of the file. You may direct IRPSIM to parse additional files by including them in your ODL file:

#include path-to-file

When IRPSIM encounters an #include directive, it parses the included file from beginning to end before continuing to parse the original file. Multiple files may be included, allowing the creation of project files.

Snippet from a project file
**Relationships**  << this is a comment
#include ..\Relationships\20x2020_580.rel
#include ..\Relationships\2013_start_year.rel

**CRA Supplies**
#include ..\Supplies\Colorado_River_Aqueduct\all_american_lining.sup
#include ..\Supplies\Colorado_River_Aqueduct\basic_apportionment.sup

#included files may be specified using relative paths or absolute paths:

relative

#include ..\Relationships\swp.rel

absolute

#include C:\\some\root\directory\Relationships\swp.rel

Relative paths of included files are calculated relative to the path of the including file.

2. IRPSIM Project File

An IRPSIM Project File, also known as a configuration file, is a file containing nothing but #include directives and optional comments. When you load a project into IRPSIM, this is the type of file it is expecting.

Example
*Script**
#include ..\Operational_Scripts\current_approach.sc

**Options**
#include ..\Options\2013_basic.opt

**Scenario**
#include ..\Scenario\current_approach.sce

**Relationships**
#include ..\Relationships\20x2020_580.rel

**CRA Supplies**
#include ..\Supplies\Colorado_River_Aqueduct\all_american_lining.sup

**In-Region Supplies**
#include ..\Supplies\In_Region\chino.sto

**SWP Supplies**
#include ..\Supplies\State_Water_Project\arvin_edison.sto

**Demands**
#include ..\Demands\retail.dem

3. Variables

Variables are the main building blocks of IRPSIM. They allow you to express the values of supply, demand, storage and other quantities used in a simulation. The basic syntax is as follows:

Basic variable syntax
#VARDEF variable_name (1)
#attribute1 value1 (2)
#attribute2 value2
...
#attributeN valueN
... declaration body ... (3)
#END (4)
1 Start of declaration
2 One or more variable attributes
3 An expression or array
4 End of declaration
User-defined variables must begin with a letter and may contain only letters, numerals and the underscore character. There is a set of IRPSIM System Variables that begin with an underscore. IRPSIM variable names are case-insensitive, so VaRiAbLe_NAME is interpreted the same as variable_name.

IRPSIM Version 9 allows two types of user-defined variable: Arrays and Expressions

3.1. Supply, Demand, Storage and Cost Variables

IRPSIM runs a simulation by manipulating supply, demand and storage variables using a script. In addition, IRPSIM keeps track of aggregate costs. In ODL, you indicate that a variable is a supply, demand, storage or cost variable by specifying a special type attribute in the variable definition.

Variable Type Attribute

Supply

#type supply

Demand

#type demand

Storage

#type storage

Cost

#type cost

3.2. Arrays

An array is a two-dimensional data structure that is indexed by a row and column. You specify an array by including #rows and #columns attributes immediately following the #VARDEF declaration:

A 3 row by 2 column array definition
#VARDEF some_array
#rows 3
#columns 2
1.3  2.4
8.5  6.7
12.0  9.2
#END

An array may be referenced in an expression by using row and column indices:

Referencing an array variable in an expressiomn
some_array[row_index][column_index]

In the example above, row_index and column_index may be constant values or expressions.

3.3. Expressions

An expression in IRPSIM is a mathematical expression that may contain operators, functions and variable references. With a few exceptions, the IRPSIM expression parser follows familiar conventions in evaluating expressions—​operator notation and precedence, nesting within parentheses, etc.

Examples of IRPSIM expressions
semitropic + arvin + mojave + kerndelta + sbvmwd_carryover + swpcarryover (1)

max(min(dwcv_put-dwcv.put,dwcv_cap-dwcv.resolved),0) (2)

if(_cyear >= start_dwcv,max(250000 - dwcv_remaining_obligation,0),0) (3)
1 A simple sum
2 Showing use of max and min functions
3 Showing use of if function and the system variable _cyear

Unlike many programming language, IRPSIM does not make use of an assignment operator, e.g. A=B. The ODL way of assigning B to the variable A is

#VARDEF A
B
#END

3.4. Variable Attributes

Variables may have one or more attributes assigned to them that affect how they are processed by IRPSIM.

Assigning Attributes to a Variable
#VARDEF variable_name
#attribute1 value1 (1)
#attribute2 value2 (2)
... declaration body ...
#END
1 The value value1 is assigned to the attribute attribute1
2 The value value2 is assigned to the attribute attribute2

3.4.1. The Type Attribute

The type attribute is IRPSIM’s most fundamental attribute. It tells IRPSIM whether the variable is a demand, supply, storage or cost variable; the stage of the variable; and whether or not to re-evaluate the variable each time it is encountered in an expression.

Examples
#VARDEF dwcv
#type storage (1)
...
#END (2)

#VARDEF hayfield
#type supply (3)
...
#ENDVAR (2)

#VARDEF mojave_take6
#type stage6 (4)
mojave.take
#END

#VARDEF cra_service_area
#type alwaysevaluate (5)
hinds_ps - dwcv.put - dwcv_remaining_obligation + dwcv_transfer_reservoir.take
#ENDVAR
1 dwcv is defined as a storage variable
2 A variable declaration may be terminated with #END or #ENDVAR
3 hayfield is defined as a supply variable
4 mojave_take6 will be evaluated when the script command "evaluate stage stage6" is executed
5 cra_service_area will be re-evaluated every time it is encountered in an expression
Type Attribute Description

demand

The declared variable is a demand variable.

supply

The declared variable is a supply variable.

storage

The declared variable is a storage variable.

cost

The declared variable is a cost variable.

alwaysevaluate

The variable will be evaluated each time it is encountered in an expression. This attribute is useful for evaluating conditional expressions in scripts that depend on the current status of puts, takes, surpluses, etc. The default behavior is for a variable evaluation to remain fixed at its initially calculated value over the course of a script execution.

stageN

A flag that determines when a variable is evaluated during the execution of a script. All variables having the stageN #type attribute will be evaluated when IRPSIM executes the script command evaluate stage stageN.
stageN is merely a naming convention for the stage execution flag. If a variable is assigned the type attribute "donkey" as in #type donkey, then the variable will be evaluated if the executing script contains the command evaluate stage donkey.
IRPSIM processes demand, supply and storage variables differently than it does "ordinary" variables, and it will ignore any other #type attributes that may be specified for these variables.
It generally doesn’t make sense for a variable to have both alwaysevaluate and a stage flag specified as #type attributes. A stage flag instructs IRPSIM to lock down the value of a variable at a particular "stage" of a script, whereas alwaysevaluate has the opposite effect: the variable will be re-evaluated if it is encountered later in the script.

3.4.2. Array Attributes

Array variables have the special attributes #rows and #columns that tell IRPSIM how many rows and columns are in the array.

Example
#VARDEF array_variable
#rows 3
#columns 2
1.3  2.4
8.5  6.7
12.0  9.2
#END

3.5. System Variables

In addition to variables that are defined by the user, IRPSIM has a number of predefined system variables that may be used in an expression. By convention, these system variables begin with an underscore (_).

3.5.1. Time- and Trial-dependent System Variables

_timestep

The current time step.

_trial

The current simulation trial.

_year

The current year (ends with the month defined in the yearend option).

_cyear

The current calendar year.

_tyear

The current trace year (ends with the month defined in the yearend property).

_ctyear

The current calendar trace year.

_period

The current period. The value depends on the siminterval of the simulation. If siminterval is annual, _period will be expressed as YYYY; if siminterval is monthly, _period will be expressed as YYYYMM, etc.

_tperiod

The current trace period. As with _period, the value depends on the siminterval of the simulation.

_month

The current month. Values range from 1 (January) to 12 (December).

_day

The current day in the month. Values range from 1 to 31.

_weekday

The current day in the week. Values range from 1 (Sunday) to 7 (Saturday)

_dayinyear

The current day in the year. Values range from 1 to 366 (during leap years)

_daysinyear

The number of days in the current year. Is either 365 or 366 (during leap years).

_daysinmonth

The number of days in the current month. Values range from 1 to 31.

_weekinyear

The current week in the year. Values range from 1 to 53 (for days 365 and 366).

_firstdayofmonth

The _dayinyear corresponding to the first day of the current month.

_lastdayofmonth

The _dayinyear corresponding to the last day of the current month.

_midmonthday

The _dayinyear corresponding to the middle day of the current month.

_jan

1 if the current month is January, 0 otherwise.

_feb

1 if the current month is February, 0 otherwise.

_mar

_apr

_may

1 if the current month is May, 0 otherwise.

_jun

1 if the current month is June, 0 otherwise.

_jul

1 if the current month is July, 0 otherwise.

_aug

1 if the current month is August, 0 otherwise.

_sep

1 if the current month is September, 0 otherwise.

_oct

1 if the current month is October, 0 otherwise.

_nov

1 if the current month is November, 0 otherwise.

_dec

_sinN

The Nth harmonic of a sine function having a period of one year, evaluated at the current _midmonthday. This value remains constant throughout a month for simulations having a siminterval of less than a month. There are six _sinN system variables: _sin1, _sin2, _sin3, _sin4, _sin5, _sin6.

_cosN

_sinNd

The Nth harmonic of a sine function having a period of one year, evaluated at the current _dayinyear. There are six _sinNd system variables: _sin1d, _sin2d, _sin3d, _sin4d, _sin5d, _sin6d.

The Nth harmonic of a cosine sine function having a period of one year, evaluated at the current _dayinyear. There are six _cosNd system variables: _cos1d, _cos2d, _cos3d, _cos4d, _cos5d, _cos6d.

3.5.2. System Aggregate Variables

IRPSIM maintains running totals of supply, demand, storage and other quantities over the course of a simulation trial. These quantites may be accessed using IRPSIM’s system aggregate variables. In addition, Regionally-qualified versions of these variables keep track of the quantities for every region that the user defines.

_cost, _cost.region

The sum of all cost variables that have been evaluated.

_demand, _demand.region

The sum of all demand variables that have been evaluated.

_demandcut, _demandcut.region

The cumulative total of all demand cuts.

_gainloss, _gainloss.region

The cumulative value of all gains to storage less the cumulative value of all losses.

_netsurplus, _netsurplus.region

Shorthand for _supply - _demand - _put + _take

_put, _put.region

The cumulative value of all puts into storage.

_storage, _storage.region

The sum of water stored in all storage devices.

_supply, _supply.region

The sum of all supply variables that have been evaluated.

_supplycut, _supplycut.region

The cumulative total of all supply cuts.

_surplus, _surplus.region

Shorthand for _supply - _demand

_take, _take.region

The cumulative value of all takes from storage.

4. Scripts

An ODL script is a detailed set of instructions that IRPSIM executes each time step of each simulation trial. A script is declared in ODL as follows:

Script Declaration
#SCRIPT script_name
* comment (1)
line 1
line 2
...
line N
#END
1 Comments begin with an asterisk

With the exception of do commands and if / endif blocks, each line of a script follows the following format:

command <modifiers>

The types of modifiers allowed depend on the command and some modifiers are optional.

4.1. Script Commands

4.1.1. Evaluate Commands

Command Allowed Modifiers Description

evaluate demand

node, category

Evaluate one or more demand nodes. This will increase the value of the _demand system variable and will decrease the values of the _surplus and _netsurplus system variables.

evaluate supply

node, category

Evaluate one or more supply nodes. This will increase the value of the _supply system variable and will increase the values of the _surplus and _netsurplus system variables.

evaluate stage

stage_name

Evaluate all variables having a type attribute of stage_name.

4.1.2. Cut Commands

Command Allowed Modifiers Description

cut demand

node, category, limit, using, ignorebalance

Cut back demand from the specified node or category using either the default allocation rule, or the allocation rule specified by the using modifier (if applied to a category). Limit the amount cut by either the amount specified in the limit clause or by net shortage (negative of _netsurplus), unless ingorebalance is specified, in which case the net shortage is not limiting. Reduces the value of the _demand system variable and increases the value of the _demandcut system variable.

cut supply

node, category, limit, using, ignorebalance

Cut back supply from the specified node or category using either the default allocation rule, or the allocation rule specified by the using modifier (if applied to a category). Limit the amount cut by either the amount specified in the limit clause or by _netsurplus, unless ingorebalance is specified, in which case _netsurplus is not limiting. Reduces the value of the _supply system variable and increases the value of the _supplycut system variable.

4.1.3. Gain and Loss Commands

Command Allowed Modifiers Description

gain

node, category

Executes natural gains to storage. If specified, the execution is limited to a node or category. Increases the value of the _storage and _gainloss system variables.

loss

node, category

Executes natural losses to storage. If specified, the execution is limited to a node or category. Decreases the value of the _storage and _gainloss system variables.

4.1.4. Put and Take Commands

Command Allowed Modifiers Description

put

node, category, limit, using, ignorebalance

Put water into the specified node or category using either the default allocation rule, or the allocation rule specified by the using modifier (if applied to a category). Limit the amount put by either the amount specified in the limit clause or by _netsurplus, unless ingorebalance is specified, in which case _netsurplus is not limiting. Increases the value of the _put, _puttake, and _storage system variables.

take

node, category, limit, using, ignorebalance

Take water from the specified node or category using either the default allocation rule, or the allocation rule specified by the using modifier (if applied to a category). Limit the amount taken by either the amount specified in the limit clause or by the net shortage (negative of _netsurplus), unless ingorebalance is specified, in which case the net shortage is not limiting. Increases the value of the _take system variable, and reduces the value of the _puttake and _storage system variable.

4.1.5. Do Command

The do command instructs IRPSIM to execute a different script. Execution of the calling script continues after the called script has executed.

The do Command
#SCRIPT CallingScript
...
do CalledScript
... (1)
#END

#SCRIPT CalledScript
...
#END
1 Execution of CallingScript continues on this line after CalledScript has finished executing

4.1.6. If / else / endif blocks

If / else / endif blocks allow you to specify a block of script that will execute only if a particular condition is satisfied.

if / endif block
if {condition} (1)
   ... block of script to execute if condition is true (evaluates to non-zero) ...
endif
if / else / endif block
if {condition} (1)
   ... block of script to execute if condition is true (evaluates to non-zero) ...
else
   ... block of script to execute if condition is false (evaluates to zero) ...
endif
1 The condition may be any valid ODL expression and it must be surrounded by curly braces {}

4.2. Command Modifiers

Modifier Syntax Description

node

node NodeName

Limits the execution of the command to the node NodeName.

category

category CategoryName

Limits the execution of the command to the category CategoryName. If the command involves an allocation, the allocation will be performed using the default allocation rule for the category, or by the allocation specified in the using modifier, if one is present.

using

using AllocationMethod

If the command involves an allocation, the allocation will be performed using the specified AllocationMethod.

limit

limit {expression}

The command operation will be limited by the expression contained within the {} braces. May be used with cut demand, cut supply, put and take commands. The expression may be any valid ODL expression and it must be surrounded by curly braces {}

ignorebalance

ignorebalance

Normally, the cut demand, cut supply, put and take commands will be constrained by the current value of _netsurplus. If ignorebalance is specified, this constraint will be ignored.

  take node sanbernardino_option

  take node Arvin

if {_netsurplus < 0}
  do shortage_script (13)

5. Scenarios

In IRPSIM Version 9, a scenario is simply a list of variables that will be included in the simulation output file:

Scenario Declaration
#SCENARIO scenario_name
variable1
variable2
variable3
...
variableN
#END
In early versions of IRPSIM, back when RAM and processing speed were limited, scenarios included optional save and write parameters that instructed IRPSIM to save variable values to binary simulation files and/or to write values to output files. The use of these parameters is deprecated, and IRPSIM will issue a warning if it encounters them when parsing scenarios.

6. Categories

Categories are used to limit certain script operations to groups of variables.

Category Declaration
#CATEGORY category_name
#allocationmethod MethodType (1)
... List of Variables ...
#END
1 The #allocationmethod attribute is optional. When it is present, MethodType must be one of the allocation methods described below.

6.1. Allocation Methods

Method Applies To Description

ByRank

Supply, Demand, Storage

IRPSIM will perform the action on each variable in the category in increasing order of the rank of each variable (as determined by the variable’s #rank attribute) until the limit of the action has been reached (e.g. all surplus water has been put).

BalancePercent

Supply, Demand

RPSIM will cut back the supply or demand of each variable in the category in proportion to each variable’s supply or demand contribution.

BalancePercentCapacity

Storage

IRPSIM will put to or take from devices in the category while attempting to balance the ratio of storage level to capacity among the devices in the category.

BalancePut

Storage

IRPSIM will put to or take from devices in the category while attempting to balance the ratio of remaining capacity to put conveyance (fullput) among the devices in the category. Loosely speaking, it tries to balance the number of years of unused capacity in each device.

BalanceTake

Storage

IRPSIM will put to or take from devices in the category while attempting to balance the ratio of storage level to take conveyance (fulltake) among the devices in the category. Loosely speaking, it tries to balance the number of years of water stored in each device.

7. Regions

Regions are not declared directly in the ODL. Rather, they are indirectly declared by assigning #region attributes to supply, demand, storage or cost variables.

Assigning a Region to a Variable
#VARDEF variable_name
#region some_region
... body of definition ...
#END

8. Definitions

A definition is an alternative to a user-defined variable that requires less memory and processing overhead than a #VARDEF variable. You may use a definition in an expression the same way you would use a variable, except that you may not use a lag index.

Example
#DEFINE DiscountRate  1.05

#VARDEF variable_name (1)
some_other_variable * DiscountRate
#END

#VARDEF variable_name (2)
some_other_variable * DiscountRate[_y-1]
#END
1 This is a valid use of DiscountRate
2 This is not a valid use of DiscountRate: definitions may not have a lag index
The #DEFINE notation may only be used to define simple, constant values. Objects expressed using this notation may not be included in simulation output files.
